# Slice 2 — X3DH session + first real encrypted message — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Retire the opaque-blob placeholder — a sender looks up a recipient, fetches their public prekey bundle, runs X3DH to establish a Signal session, and exchanges a genuinely E2EE message that the recipient decrypts; the server sees only ciphertext + minimal routing metadata.

**Architecture:** Contract-first, then TDD both sides. Server (hexagonal, Testcontainers-heavy) gains an exact-match user-lookup endpoint and an atomic one-time-prekey bundle-fetch endpoint, and the message path is reconciled to integer libsignal device numbers + an explicit `messageType`. iOS (Swift Testing against mocks, real libsignal on-device) gains a `SessionStore`, an X3DH `SessionEstablisher`, a `MessageCrypto`, two REST clients, and start-conversation + chat-thread screens. The chat thread is ephemeral (in-memory) this slice; persisted history is Slice 3.

**Tech Stack:** Java 25 / Spring Boot 4, JPA + Flyway, Confluent Avro on Kafka, Testcontainers (Postgres + Keycloak), Spring Security resource server; Keycloak Admin REST API. Swift 6 / SwiftUI, LibSignalClient 0.96.2 (CocoaPods), GRDB + SQLCipher, AppAuth; Swift Testing.

## Global Constraints

- **Privacy (root CLAUDE.md principle 6):** server stays content-blind — only `ciphertext` (opaque), routing identities (`sub`, integer device number), `messageId`, and `messageType` may be cleartext; each is justified in the contracts. No plaintext/keys/tokens logged. Private keys **and session state** never leave the iOS SQLCipher vault.
- **TDD red→green→refactor** on every change. Server inner loop = `@SpringBootTest` Testcontainers integration tests (no mocked sockets). iOS inner loop = Swift Testing against mocks/fakes, no backend, no Docker.
- **Coverage ≥ 90%** both sides (JaCoCo server; Xcode coverage iOS — views/app-entry/CocoaPods/platform-edge adapters excluded from the denominator).
- **Lint must pass / fails the build:** Checkstyle (Google) on the server; SwiftLint strict on iOS (no force-unwraps — use `#require`; no `String(decoding:as:)` — use `String(bytes:encoding:)`).
- **Contract source of truth** is `docs/contracts/` (server is authority); iOS fixtures are copied into the test bundle so the mock can't drift.
- **Recipient lookup is exact-match only** — no listing/prefix/autocomplete/enumeration; rate-limited.
- **Slice boundary:** persisted/restored history, Double-Ratchet across many messages, and out-of-order tolerance are **Slice 3** — do not build them here. Multi-device (device numbers > 1) is **Slice 5**; `deviceId` is always `1`. One-time Kyber prekeys + prekey rotation/replenishment are **Slice 9**.
- **PQXDH (mandatory — Task 0 verdict):** every prekey bundle carries a **last-resort signed Kyber (ML-KEM-1024) prekey** — generated on-device, identity-signed, published, stored, served at fetch, never consumed (mirrors the signed prekey). It is **mandatory** in the bundle (pre-release; no back-compat — the two Slice-1 test users re-register). See `docs/superpowers/specs/2026-06-27-pqxdh-amendment-design.md`. Exact serialized `KEMPublicKey` / signature byte-lengths are **pinned during implementation** (read from a generated key, as Slice 1 pinned EC pubkey=33, sig=64) — never guessed.
- **Migrations:** the message-envelope reconciliation needs **no** migration (B1); the Kyber prekey storage needs **Flyway `V3`** (Task BK).
- **iOS env (from Slice 1):** `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer`; build the **workspace** (`xcodegen generate && pod install` → `Cloak.xcworkspace`); pre-boot the simulator (`iPhone 17`) before `xcodebuild test`.
- **Branch:** `feature/slice-2-x3dh` (already created; the spec commit is on it). Squash+rebase into one consolidated commit before the PR.

> **Verified API facts (0.96.2):** `PreKeyBundle.init` requires `kyberPrekeyId:kyberPrekey:kyberPrekeySignature:` (two overloads — with/without one-time prekey; both need Kyber). `processPreKeyBundle(_:for:ourAddress:sessionStore:identityStore:now:context:)`. `signalEncrypt(message:for:localAddress:sessionStore:identityStore:now:context:)`. `signalDecrypt(message:from:to:sessionStore:identityStore:context:)`. `signalDecryptPreKey(message:from:localAddress:sessionStore:identityStore:preKeyStore:signedPreKeyStore:kyberPreKeyStore:context:)`. `CiphertextMessage.MessageType` is a **struct** (not enum) — compare with `CiphertextMessage.MessageType.preKey` / `.whisper` (raw 3 / 2). `KEMKeyPair.generate()`, `NullContext`, and `InMemorySignalProtocolStore` (conforms incl. `KyberPreKeyStore`) all exist.

**Why first:** LibSignalClient 0.96.2 is PQXDH-era. If `PreKeyBundle`/`processPreKeyBundle` *require* Kyber prekey parameters, then Slice 1's published bundle (identity + signed + one-time only) is insufficient and Slice 2's scope grows to include Kyber-prekey generation (iOS), the bundle contract, and server storage (a Flyway migration). This task confirms the exact installed signatures **before** any production code is written, so the rest of the plan is built on verified API facts (not assumptions). No production code ships from this task.
